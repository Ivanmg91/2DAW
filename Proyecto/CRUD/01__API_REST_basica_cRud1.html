NIVEL 1 – Traza completa de una API REST básica

Qué vamos a hacer

Un endpoint:

GET /api/users

Que devuelve usuarios en JSON.

0️⃣ Estado inicial (suposiciones)

Base de datos (users)

id name email
1 Ana ana@mail.com
2 Luis luis@mail.com

1️⃣ ENTRADA – Petición HTTP

El cliente (Postman, navegador, frontend) envía:

GET http://miapp.test/api/users

  • Esto es lo único que hace el cliente.

2️⃣ Route – ¿A dónde va esta petición?

Archivo: routes/api.php

Route::get('/users', [UserController::class, 'index']);

Qué pasa aquí internamente

Laravel piensa:

“Ok, alguien pidió /api/users,
tengo que llamar a UserController@index”

  • No ejecuta lógica
  • No toca la base de datos
  • No valida nada

-️ Solo redirige

3️⃣ Controller – Punto de entrada del backend

Archivo: app/Http/Controllers/UserController.php

use App\Models\User;

class UserController extends Controller
{
    public function index()
    {
        return User::all();
    }
}

Qué hace el controller

  1. Llama al modelo User (tabla de la DB)
  2. Pide todos los registros
  3. Devuelve el resultado
  • El controller NO sabe SQL,
  • NO formatea JSON,
  • NO valida nada

4️⃣ Model (Eloquent) – Habla con la base de datos

Archivo: app/Models/User.php

class User extends Model
{
}

Cuando ejecutas:

User::all();

Laravel internamente hace algo como:

SELECT * FROM users;
  • Eloquent convierte filas → objetos PHP
[
  User { id: 1, name: "Ana", email: "ana@mail.com" },
  User { id: 2, name: "Luis", email: "luis@mail.com" }
]

5️⃣ Respuesta automática de Laravel

Como no devolvimos una vista, Laravel hace esto:

  • Detecta que es una API
  • Convierte los modelos a JSON
  • Añade headers correctos

1️⃣ La ruta no ejecuta lógica

Preguntas varias:

1️⃣ ¿Cómo sabe Laravel a QUÉ base de datos conectarse?

Archivo clave

`.env`

Ejemplo típico:

DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=mi_base_datos
DB_USERNAME=root
DB_PASSWORD=secret`

Qué hace Laravel con esto

  1. Laravel lee el .env
  2. Lo carga en config/database.php
  3. Crea la conexión al arrancar la app
  • Tú nunca llamas a la conexión manualmente
  • Laravel la mantiene lista

Flujo mental

`.env → config/database.php → conexión activa`

Cuando haces:

`User::all();`

Laravel ya sabe:
-️ motor
-️ host
-️ base de datos
-️ credenciales

2️⃣ ¿Cómo sabe Laravel QUÉ tabla usar?

Aquí entra Eloquent (el ORM).

Un ORM (Object-Relational Mapping o Mapeo Objeto-Relacional) es una técnica de programación que funciona como un “traductor” entre el código orientado a objetos (como PHP o Python) y las bases de datos relacionales (como MySQL o PostgreSQL).
¿Qué es exactamente un ORM?
En lugar de escribir consultas SQL complejas a mano, un ORM te permite interactuar con los datos usando los mismos objetos con los que programas.
-Tablas se convierten en Clases.
- Filas se convierten en Objetos (instancias de esas clases).
-Columnas se convierten en Propiedades del objeto.
.
¿Qué es Eloquent?
Eloquent es el ORM que viene incluido por defecto en el Framework Laravel. Se distingue por ser:
.
-Active Record: Implementa este patrón donde cada modelo (clase) representa una tabla y es responsable de sus propias operaciones de base de datos.
-Intuitivo: Su sintaxis es casi como leer inglés. Por ejemplo, para obtener todos los usuarios, solo escribes User::all();.
-Gestor de Relaciones: Facilita enormemente el manejo de relaciones complejas como uno a muchos o muchos a muchos sin necesidad de JOINs manuales.

Regla de oro de Eloquent

El nombre del modelo determina el nombre de la tabla

Convención:

Modelo Tabla
User users
Post posts
Category categories
OrderItem order_items
  • Singular → plural
  • CamelCase → snake_case

Ejemplo real

Modelo

app/Models/User.php

class User extends Model
{
}

Laravel asume automáticamente:

SELECT * FROM users;

Sin que tú escribas nada.

3️⃣ ¿Y si la tabla NO se llama así?

Entonces se lo dices explícitamente.

class  User  extends  Model { 
    protected  $table = 'mis_usuarios';
}

Ahora:

User::all();`

ejecuta:

SELECT  *  FROM mis_usuarios;`

4️⃣ ¿Cómo sabe QUÉ columnas existen?

Laravel NO lo sabe previamente.

  • Ejecuta el SELECT
  • MySQL devuelve las columnas
  • Eloquent crea atributos dinámicos
$user->email;
$user->created_at;

5️⃣ ¿Y el ID / clave primaria?

Por defecto Laravel asume:

PRIMARY KEY = id

Si es diferente:

class User extends Model
{
    protected $primaryKey = 'user_id';
}

6️⃣ ¿Y si la clave NO es autoincremental?

class User extends Model
{
    public $incrementing = false;
    protected $keyType = 'string';
}

7️⃣ Resumen ultra claro

Cuando ejecutas:

User::all();

Laravel sigue esta lógica:

1️⃣ Usa la conexión definida en .env
2️⃣ Usa la tabla users (por convención)
3️⃣ Usa id como clave primaria
4️⃣ Ejecuta el SELECT
5️⃣ Convierte filas → objetos
